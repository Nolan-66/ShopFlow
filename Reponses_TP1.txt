Q1.1 - Résultat de pytest --collect-only
------------------------------

La commande pytest --collect-only s’exécute correctement et permet de vérifier que pytest détecte les tests sans les exécuter.

------------------------------
Q1.2 - Différence entre scope="function" et scope="module"
------------------------------

Le scope="function" signifie que la fixture est recréée pour chaque test. Cela garantit un isolement total entre les tests, car chaque test utilise une base de données et une session propres.

Le scope="module" signifie que la fixture est créée une seule fois pour tout le fichier de test. Tous les tests du module partagent alors la même ressource.

Dans ce TP, le scope="function" est utilisé pour éviter les effets de bord entre les tests et garantir leur indépendance.

------------------------------
Q1.3 - Rôle de session.rollback()
------------------------------

La méthode session.rollback() permet d’annuler toutes les modifications effectuées pendant un test.

Elle garantit que les données créées ou modifiées lors d’un test ne persistent pas pour les tests suivants. Cela permet d’éviter les interférences entre tests et assure des résultats fiables et reproductibles.

------------------------------
Q1.4 - Contenu du fichier .coveragerc
------------------------------

Le fichier .coveragerc permet de configurer Coverage.py.

Exemple utilisé :

[run]
source = app
omit = */migrations/*, */__init__.py, */conftest.py

[report]
fail_under = 80
show_missing = true

[html]
directory = htmlcov
title = ShopFlow — Coverage Report

Ce fichier permet de :
- limiter l’analyse au dossier app
- exclure certains fichiers non pertinents
- afficher les lignes non couvertes
- fixer un seuil minimal de couverture

------------------------------
Q3 - Mocking Redis
------------------------------

Le mocking permet de simuler un comportement sans dépendre d’un service externe réel.

Dans ce TP, Redis est mocké avec pytest-mock afin de tester les fonctions de gestion du stock sans nécessiter un serveur Redis.

Cela permet de :
- tester le code indépendamment de Redis
- vérifier que certaines fonctions (delete, set) sont appelées
- éviter les dépendances externes
