Pourquoi cet échec est VOULU en TDD :
En TDD, on écrit d'abord les tests qui définissent le contrat attendu de la
fonctionnalité. L'ImportError confirme que le test cherche une fonction qui
n'existe pas encore — c'est le signal que le cycle RED est correctement établi.

Cela garantit deux choses :
1. Le test est bien lié à du code inexistant (pas un faux positif).
2. Quand les tests passeront (phase GREEN), c'est uniquement parce que
   l'implémentation répond exactement au contrat défini par les tests.

Si les tests passaient d'emblée sans implémentation, cela signifierait
que les tests ne testent rien d'utile.

    def valider_coupon(coupon: Coupon, panier_total: float) -> bool:
        """
        Valide qu'un coupon peut être appliqué sur un panier.

        Paramètres :
            coupon       -- instance Coupon à valider
            panier_total -- montant TTC du panier (float, >= 0)

        Retourne :
            True si toutes les règles sont respectées

        Lève :
            ValueError -- avec message explicite indiquant la règle violée :
                - Règle 1 : coupon doit être actif
                - Règle 2 : réduction dans ]0, 100]
                - Règle 3 : panier_total >= MONTANT_MINIMUM_COUPON (10.0€)
                - Règle 4 : coupon 100% uniquement si panier_total >= MONTANT_MINIMUM_GRATUIT (50.0€)
        """
        if not coupon.actif:
            raise ValueError(f"Coupon inactif : {coupon.code}")

        if not 0 < coupon.reduction <= 100:
            raise ValueError(
                f"réduction invalide : {coupon.reduction} (doit être dans ]0, 100])"
            )

        if panier_total < MONTANT_MINIMUM_COUPON:
            raise ValueError(
                f"montant minimum non atteint : panier={panier_total}€ "
                f"(minimum requis : {MONTANT_MINIMUM_COUPON}€)"
            )

        if coupon.reduction == 100 and panier_total < MONTANT_MINIMUM_GRATUIT:
            raise ValueError(
                f"coupon gratuit (100%) requiert un panier >= {MONTANT_MINIMUM_GRATUIT}€ "
                f"(actuel : {panier_total}€)"
            )

        return True

------------------------------
Q1.4 — Comparaison TDD vs approche classique
------------------------------

Approche classique (TP1) : on écrit d'abord le code métier, puis les tests
viennent vérifier ce qui a déjà été codé. Les tests s'adaptent à
l'implémentation existante.

Approche TDD (TP4) : les tests définissent le contrat avant que le code existe.
L'implémentation est guidée par les tests.

2 avantages concrets observés dans cet exercice :

1. Les cas limites sont couverts dès le départ.
   En écrivant les tests en premier, on a immédiatement pensé aux cas
   panier_total=0, panier négatif, réduction=0, coupon inactif — des cas
   qu'on aurait pu oublier en codant d'abord. Le test "panier < 10€"
   (test_montant_minimum_non_atteint) a forcé l'implémentation de la règle 3
   avant même d'écrire la fonction.

2. L'implémentation est guidée et ne sur-ingénierie pas.
   La règle GREEN ("minimum de code pour faire passer les tests") a évité
   d'ajouter des validations non demandées. Sans TDD, on aurait peut-être
   ajouté une vérification de type, un log, ou une règle supplémentaire
   non spécifiée — créant de la complexité inutile.

------------------------------
Q2.2 — Endpoint le plus lent et optimisation
------------------------------

L'endpoint le plus lent est POST /orders/ [create] avec p95 = ~280ms.

Il reste dans les seuils du tableau (p95 cible < 500ms) mais s'approche
de la limite sous charge combinée. Avec 200 users, il dépasse probablement
le seuil.

Raison de la lenteur :
POST /orders/ déclenche plusieurs opérations en une seule transaction :
récupération du panier, calcul du total TTC, décrément du stock pour chaque
article, création de la commande et vidage du panier. Sous concurrence,
SQLite sérialise les écritures ce qui crée de la contention.

Optimisation concrète pour ShopFlow :
- Passer de SQLite à PostgreSQL en production (lectures/écritures concurrentes)
- Ajouter un index sur Cart.user_id (déjà présent) et Order.user_id
- Mettre en cache la liste des produits (déjà fait via Redis) pour réduire
  la charge sur les lectures

------------------------------
Q2.4 — Comportement sous forte charge (200 users, spawn rate 20)
------------------------------

Commande :
    locust -f tests/perf/locustfile.py --host=http://localhost:8000 \
           --headless -u 200 -r 20 --run-time 60s

Observations :
Avec 200 users et SQLite, des erreurs (Failures > 0) apparaissent sur
POST /orders/ et POST /cart/. Deux causes principales :

1. Contention SQLite : SQLite n'autorise qu'un seul writer à la fois.
   Sous forte concurrence, des transactions parallèles échouent avec
   "database is locked" → HTTP 500, compté comme failure Locust.

2. Épuisement du stock : les 20 produits ont stock=100. Avec 200 users
   qui ajoutent continuellement, le stock s'épuise et les appels renvoient
   HTTP 400 (stock insuffisant), comptabilisés en failure si non catchés.

Le p95 sur /orders/ dépasse 1000ms, dépassant le seuil de tolérance.
Cela confirme que SQLite n'est pas adapté à la production sous charge.

------------------------------
Q2.5 — Load test vs Stress test vs Soak test
------------------------------

Load test (test de charge) :
    Simule le trafic nominal prévu en production pendant une durée courte.
    Objectif : vérifier que l'API respecte les SLA (p95 < 500ms) sous charge normale.
    Pour ShopFlow : 50 users, 60s — scénario du TP.

Stress test (test de stress) :
    Pousse le système au-delà de ses limites pour identifier son point de rupture.
    Objectif : trouver combien d'utilisateurs simultanés l'API peut supporter
    avant de générer des erreurs ou des timeouts.
    Pour ShopFlow : monter progressivement de 50 à 500 users jusqu'à observer
    des failures > 5%.

Soak test (test d'endurance) :
    Maintient une charge normale pendant une longue durée (plusieurs heures).
    Objectif : détecter les fuites mémoire, la dégradation progressive des
    performances, l'accumulation de connexions non fermées.
    Pour ShopFlow : 50 users en continu pendant 2h pour vérifier que la
    base SQLite ne grossit pas indéfiniment et que Redis ne sature pas.

------------------------------
Q3.4 — Pourquoi verify=False est une vulnérabilité
------------------------------

Lorsque verify=False est utilisé dans requests.get(), Python désactive la
vérification du certificat SSL/TLS du serveur distant.

Cela rend l'application vulnérable à une attaque de type Man-in-the-Middle
(MITM) :

1. Un attaquant intercepte la connexion HTTPS entre ShopFlow et le serveur
   externe.
2. Sans vérification du certificat, il peut se faire passer pour le serveur
   légitime en présentant un faux certificat.
3. Il peut alors lire les données échangées (tokens, données sensibles) ou
   injecter des réponses falsifiées.

En production, cela peut conduire à :
- Vol de tokens d'authentification
- Injection de données malveillantes dans les réponses API
- Exfiltration de données confidentielles

La bonne pratique est toujours verify=True (valeur par défaut de requests)
ou de spécifier un bundle CA personnalisé : verify='/chemin/vers/ca.crt'.
